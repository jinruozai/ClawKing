# 龙虾之王 (ClawKing) - 游戏设计方案

## 1. 项目概述

**龙虾之王**是一款基于区块链智能合约的竞争性策略游戏，完全依托智能合约和GitHub Pages部署，无需服务器。玩家通过设计龙虾AI策略和选择卡牌组合，在生成的战棋地图上进行自动战斗，争夺胜利并赢取奖池奖励。

### 核心亮点
- 战棋式自动战斗系统
- 完全基于智能合约的公平结算
- 自定义龙虾AI策略
- 低Gas费网络（Polygon等）
- 无需服务器，GitHub Pages部署
- 门票抽成商业模式
- NFT龙虾交易潜力

## 2. 游戏流程

### 2.1 匹配流程
1. **选择竞技场**：玩家选择要参与的竞技场等级（初级、中级、高级）
2. **匹配机制**：
   - 点击匹配按钮后，系统自动寻找对应等级竞技场的空闲房间
   - 如果有空闲房间，直接加入
   - 如果没有空闲房间，创建新房间
3. **多房间参与**：玩家可以同时加入多个房间（上限100个）

### 2.2 房间界面
- **左侧列表**：显示玩家所有加入的房间
- **右侧主视图**：显示当前选中房间的详细内容

### 2.3 开局准备
1. **凑局**：每个房间凑够对应人数（具体人数由竞技场规则决定）
2. **门票**：根据竞技场等级支付对应门票（具体价格由竞技场规则决定）
3. **地图生成**：基于合约哈希值和地图等级生成房间地图
   - 包含不同地形和障碍物
   - 地图上随机分布武器/道具/技能（引用合约表中的ID）
4. **龙虾数据上传**：
   - 图片链接
   - 属性分配（总属性值固定，玩家自由分配）：
     - 生命值
     - 魔法值（使用技能消耗）
     - 攻击
     - 防御
     - 移动速度（决定每回合最大移动距离）
   - 背包大小
   - 策略脚本（自定义AI逻辑）
5. **初始位置选择**：玩家在战棋地图上选择出生位置
6. **卡牌选择**：
   - 从公牌中选择若干张卡牌（数量根据背包尺寸）
   - 卡片拥挤规则：若某张卡片被N(≥2)只龙虾同时选择，该卡片属性削弱为原属性/N
7. **提交**：将选择结果提交到智能合约

### 2.4 对战流程
1. **地图加载**：根据地图等级和哈希值生成地图
2. **位置分配**：根据玩家选择的初始位置放置龙虾
3. **战斗机制**：按回合进行战棋战斗
   - 每回合调用玩家策略脚本获取行动
   - 行动包括：移动、攻击、使用道具/技能、拾取物品等
   - 攻击计算：攻击力 - 目标防御力 + 武器/道具效果
   - 魔法消耗：使用技能时消耗对应魔法值
   - 地形影响：不同地形提供不同加成或减益
   - 物品交互：拾取地图上的武器/道具/技能
4. **胜利条件**：最后存活的龙虾获胜
5. **奖励分配**：根据规则分配奖池

## 3. 战棋系统

### 3.1 地图设计
- **地图生成**：基于合约哈希值和地图等级
- **初级竞技场**：32×32网格，基础地形
- **中级竞技场**：64×64网格，包含简单障碍物和地形
- **高级竞技场**：128×128网格，复杂地形和动态元素

### 3.2 地形系统
- **普通地形**：无特殊效果
- **高地**：攻击范围+1，防御+10%
- **水域**：移动消耗增加，攻击速度-10%
- **障碍物**：无法通过，可作为掩护
- **能量节点**：站在上面的龙虾获得临时buff

### 3.3 物品系统
- **武器**：有耐久度，增加攻击力
- **道具**：消耗品，提供各种效果
- **技能**：可无限使用，消耗魔法值

### 3.4 AI策略系统
- **模板策略**：系统提供多种预设AI策略
- **自定义策略**：玩家可编写脚本定义龙虾行为
- **策略接口**：每回合返回action及参数
- **行动类型**：移动、攻击、使用道具、使用技能、拾取物品

## 4. 数据表系统

### 4.1 规则表
- 定义不同竞技场的规则
- 包含奖池分配、地图尺寸、参与条件等

### 4.2 武器表
- 武器ID、名称、攻击力、耐久度、特殊效果
- 不同品质和类型的武器

### 4.3 道具表
- 道具ID、名称、效果、使用次数
- 消耗品，使用后消失

### 4.4 技能表
- 技能ID、名称、效果、魔法消耗
- 可无限使用，需要消耗魔法值

## 5. 经济系统

### 5.1 门票与奖池
- 门票：根据场次不同，价格从0.5刀到10刀不等
- 奖池：N × 门票（N为参赛人数）
- 平台抽成：固定12.5%
- 分配：平台抽成后，剩余部分按规则表设定分配给获胜者

### 5.2 Gas费覆盖方案
- **Gas费来源**：平台抽成部分优先用于覆盖Gas费
- **Gas费覆盖分析**：
  - 以8人局为例，门票1刀/人，总门票8刀，平台抽成1刀
  - Polygon网络Gas费较低，每笔交易约0.01-0.1刀
  - 主要Gas消耗：
    - 房间创建与门票收取：8笔交易 × 0.05刀/笔 = 0.4刀
    - 战斗结果计算：约1-2笔交易 × 0.1刀/笔 = 0.1-0.2刀
    - 结果结算与奖励发放：1笔交易 × 0.1刀/笔 = 0.1刀
    - 排行榜更新：1笔交易 × 0.05刀/笔 = 0.05刀
  - 总Gas费约：0.65-0.75刀
  - **结论**：12.5%抽成（1刀）足够覆盖Gas费，即使Gas费波动上涨，仍能保证平台有合理收益

- **抽成比例合理性**：
  - 12.5%抽成既能覆盖Gas费波动，又能为平台带来合理收益
  - 与行业标准（5-20%）相比，处于合理范围
  - 随着玩家数量增加，规模效应会降低单位Gas成本，平台收益会相应提升

### 5.3 竞技场设置
根据玩家等级和门票价格，可设置不同竞技场等级：
- **初级竞技场**：
  - 门票：1刀
  - 每房间人数：8人
  - 地图尺寸：32×32
  - 解锁条件：所有等级可参与
- **中级竞技场**：
  - 门票：5刀
  - 每房间人数：16人
  - 地图尺寸：64×64
  - 解锁条件：黄金及以上段位
- **高级竞技场**：
  - 门票：10刀
  - 每房间人数：32人
  - 地图尺寸：128×128
  - 解锁条件：铂金及以上段位

**可扩展性**：后续可根据需要添加更多类型的竞技场，只需在规则表中定义即可

### 5.4 房间机制
- 龙虾报名花费门票加入房间后，房间会持续锁定24小时等待其他龙虾加入
- 凑够对应人数后立即开始比赛（初级竞技场8人，中级竞技场16人，高级竞技场32人）
- 如果24小时没有凑够人数，则房间合约解散，门票原路返回（可能会有gas消耗）
- 解散的房间平台不会抽成

### 5.5 房间生成机制
- **自动生成**：当玩家点击匹配且对应等级竞技场无空闲房间时，自动创建新房间
- **房间上限**：每个玩家最多可加入100个房间
- **房间状态**：前端实时显示房间状态（等待中、进行中、已结束）
- **匹配逻辑**：智能合约自动匹配玩家到合适的房间，优先填满已有房间

## 6. 等级与段位系统

### 6.1 等级系统
- **等级获取**：每局比赛结束后，龙虾根据表现和结果获得经验，经验满了会升级
- **等级作用**：类似英雄联盟等游戏，主要用于解锁皮肤，没有实际功能
- **等级划分**：从1级开始，无上限

### 6.2 段位系统
- **段位获取**：通过积分决定段位，赢了加分，失败扣分
- **段位保护**：
  - 每个段位有积分基底（例：白银段位90分）
  - 当积分接近段位基底时，连续失败三次才会真正掉段
  - 例：玩家100积分（白银段位），输一次扣8分→92分
  - 再输一次，扣到段位基底90分（触发保底）
  - 再输一次，保持90分（继续保底）
  - 连输两次后，再输一次，才会掉到90分以下，真正掉段
- **段位划分**（从低到高）：
  1. 黑铁
  2. 青铜
  3. 白银
  4. 黄金
  5. 铂金
  6. 翡翠
  7. 钻石
  8. 大师
  9. 宗师
  10. 王者

### 6.3 竞技场解锁
- 竞技场解锁一般依赖段位
- 具体解锁条件在各竞技场定义中指定
- 段位越高，可参与的竞技场种类越多，潜在奖励也越高

### 6.4 数据存储
- 龙虾数据（等级、经验、段位、积分、战绩、皮肤、AI策略）存储在链上
- 每个龙虾拥有唯一的链上身份
- 数据可通过智能合约查询和验证

### 6.5 排行榜
- 全球前一千名排行榜，基于龙虾的总积分
- 排行榜数据存储在链上，确保透明公正
- 定期更新，展示最顶尖的龙虾玩家

### 6.6 皮肤系统
- **皮肤获取**：
  - 等级解锁皮肤
  - 段位皮肤（使用后会随着段位变化自动更换）
  - 特殊奖励皮肤
- **皮肤功能**：仅展示，没有任何实际功能属性
- **皮肤使用**：每个龙虾都可以使用不同的皮肤

## 7. NFT系统

### 7.1 龙虾NFT

- **唯一性**：每个NFT龙虾都有唯一的链上身份和属性

3. **透明度**：所有游戏过程和结果可在链上查询
4. **低成本**：利用低Gas费网络，降低玩家参与成本
5. **可扩展性**：可通过智能合约升级添加新内容
6. **链上数据**：玩家数据和排行榜完全存储在链上，确保数据安全和透明

## 11. 技术可行性分析

### 11.1 无需服务器的实现方案

**完全可以实现不用服务器，具体方案如下：**

1. **前端**：
   - 使用GitHub Pages托管静态网站
   - 所有前端逻辑在浏览器中执行
   - 通过Web3.js/ethers.js与智能合约交互

2. **智能合约**：
   - 部署在Polygon等低Gas费网络
   - 处理所有核心游戏逻辑：
     - 门票收取与奖池管理
     - 房间创建与管理（24小时倒计时）
     - 地图生成（基于哈希值和等级）
     - 战斗结果计算
     - 结果结算与奖励发放
     - 玩家数据存储（等级、经验、段位、积分、战绩、皮肤、AI策略）
     - 排行榜数据维护
     - NFT龙虾管理

3. **数据存储**：
   - 所有游戏数据存储在链上
   - 玩家数据通过智能合约映射存储
   - 排行榜数据通过智能合约维护

4. **随机数生成**：
   - 使用链上VRF（可验证随机函数）生成随机元素
   - 确保随机性和公平性

5. **时间管理**：
   - 使用智能合约的block.timestamp实现24小时房间倒计时
   - 无需外部服务器触发

### 11.2 优势与挑战

**优势：**
- 零服务器成本
- 完全去中心化，公平公正
- 数据透明，不可篡改
- 全球可访问

**挑战：**
- 智能合约开发复杂度较高
- Gas费管理（需优化合约代码减少Gas消耗）
- 前端需处理链上数据的读取和展示
- 策略脚本的链上执行效率

### 11.3 排行榜链上部署分析

**排行榜可以免费部署在链上，但需要注意以下几点：**

1. **存储成本**：
   - 链上存储数据需要Gas费，但可以通过优化存储结构来降低成本
   - 只存储前1000名的数据，减少存储量
   - 使用高效的数据结构（如映射和数组结合）

2. **更新成本**：
   - 每次比赛结束更新排行榜会产生Gas费
   - 可以采用批量更新策略，减少交易次数
   - 优化排序算法，降低计算复杂度

3. **实现方案**：
   - 使用智能合约的映射存储玩家积分
   - 维护一个排序数组存储前1000名玩家
   - 每次比赛结束后，只更新涉及的玩家排名

4. **成本控制**：
   - 利用Polygon等低Gas费网络
   - 优化合约代码，减少Gas消耗
   - 合理设计排行榜更新频率

**结论**：排行榜完全可以在链上免费部署，通过合理的设计和优化，可以控制Gas费在可接受范围内。
